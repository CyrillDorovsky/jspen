Running JSLint from the Command Line

Submitted by Paul Venuti on Mon, 2011-01-10 18:48
Tags:
  • javascript

While working on a recent project involving an embeddable JavaScript widget, I discovered a lot of useful techniques for creating widgets and for working with JavaScript in general. One of the latter is to use Douglas Crockford's JSLint to syntax- and quality-check your code.

Without a tool like JSLint, the only way to really check your code for syntax errors is to run it in every browser you plan to support. Each browser has its own set of syntactical peculiarities, most of which are of the small-and-retiring variety, rather than of the big-and-honking kind.

A good example of the kind of hard-to-track-down error that exists in the browser-based world of JavaScript is the use of a comma after the last element in an array. Adding a comma after the last element of an array is considered something of a best practice in PHP, but is not valid JavaScript syntax. Firefox, Chrome, and most other browsers will look the other way on this one, but IE pukes up a useless error message – the never-helpful "unexpected identifier, string or number" (which – applying the rules of our good friend DeMorgan – boils down to "unexpected something"). This is precisely the kind of thing that JSLint will catch.

Clearly, then, using JSLint is The Best And Right Thing To Do. But this is JavaScript, 99% of which is run in the browser; as such, JSLint only exists online. In practice, this means copying and pasting all your JavaScript into JSLint every time you want to run a syntax or quality check (referred to as "linting" your code). That gets old so fast that you probably won't end up linting your code at all.

Luckily, some folks in the JavaScript community have found a way to run JSLint from the command line! Here's what you want to do:

  1. Use macports or homebrew to install spidermonkey. SpiderMonkey is Mozilla's JavaScript engine broken out as a C program. Installing SpiderMonkey basically gives you a JavaScript shell, which is exactly what we need if we want to run JSLint from the command line.

    Normally I use homebrew to build from source, but it had troubles with the spidermonkey install, so I'd suggest using macports for this one:

    port install spidermonkey

  2. Get Andy Walker's "SpiderMonkey hack version" of JSLint. You need Walker's version of JSLint because JSLint can't quite be run from a CLI out of the box. Thankfully, Walker has made all the necessary tweaks. You can read all about the tweaks and why we need to make them on Walker's blog, but you're not missing much if you don't.

  3. Once you've downloaded the hacked version of JSLint, open it up and make sure the shebang points to your copy of SpiderMonkey. By default, Walker's version of JSLint has SpiderMonkey living at /usr/bin/js, but yours might not live there. To find out where your copy lives, run the following command:

    which js

    And change the first line of JSLint to that location.

    #!/path/to/js

  4. Put JSLint somewhere in your PATH. I put mine in /usr/bin, but anywhere in your PATH will do.

  5. That's it! You're ready to run JSLint from the command line. All you have to do is pipe your JavaScript file to the jslint command, like so:

    cat my_awesome_widget.js | jslint

Awesome!

As of now, the only drawback is that there's no way to pass in command line arguments, so what you get is the "base" JSLint – it's like running the online version without any of the options turned on. Even so, if using JSLint from the command line means that you'll run it early and often, then it's well worth it.
